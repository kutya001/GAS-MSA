# **Functional Specification Document (FSD)**

## **Руководство по внедрению DMD-конструктора в сторонние SaaS-системы (ERP, CRM, WMS, PIM)**

### **1\. Введение**

Настоящий документ описывает функциональные спецификации, логическую архитектуру и структуры данных для внедрения модуля управления мастер-данными (Master Data Management — MDM). Модуль предоставляет динамический интерфейс (Entity Builder) для создания и управления сущностями со свободно конструируемым набором характеристик без необходимости изменения физической схемы базы данных.

### **2\. Архитектурный подход**

В основе системы лежит гибридная модель **EAV (Entity-Attribute-Value)** с использованием документоориентированного подхода (хранение значений в формате JSON/Key-Value).

Архитектура разделена на три логических слоя:

1. **Справочники (Dictionaries):** Изолированные списки значений, обеспечивающие ссылочную целостность и переиспользование данных (например, справочник «Бренды», «Единицы измерения»).  
2. **Шаблоны (Templates):** Метаданные (схемы), описывающие классы объектов (например, «Смартфоны», «Одежда»). Шаблон содержит массив атрибутов, определяющих типы данных, правила валидации и формулы.  
3. **Номенклатура/Сущности (Products):** Экземпляры данных, привязанные к конкретному шаблону. Хранят фактические значения динамических атрибутов в виде единого JSON-объекта (или Map/Dictionary в зависимости от СУБД).

### **3\. Логическая структура базы данных**

Структура спроектирована без жесткой привязки к реляционным (SQL) или нереляционным (NoSQL) СУБД. Описаны абстрактные сущности и их связи.

#### **3.1. Сущность Dictionaries (Справочники)**

Хранит одномерные списки для использования в атрибутах ссылочного типа.

* **id** \[String/UUID\] \- Уникальный идентификатор справочника (Primary Key).  
* **name** \[String\] \- Человекочитаемое наименование (напр., "Цвета", "Производители").  
* **items** \[Array of Strings\] \- Массив строковых значений.

#### **3.2. Сущность Templates (Шаблоны)**

Определяет категорию или класс номенклатуры.

* **id** \[String/UUID\] \- Уникальный идентификатор шаблона (Primary Key).  
* **name** \[String\] \- Наименование шаблона.  
* **description** \[String, Nullable\] \- Подробное описание назначения шаблона.

#### **3.3. Сущность Attributes (Атрибуты шаблона)**

Описывает схему полей для шаблона. В реляционных БД реализуется как отдельная таблица со связью One-to-Many к Templates. В NoSQL БД может быть вложенным массивом внутри документа Templates.

* **id** \[String/UUID\] \- Уникальный идентификатор атрибута (Primary Key).  
* **templateId** \[String/UUID\] \- Ссылка на родительский шаблон (Foreign Key).  
* **name** \[String\] \- Наименование атрибута (напр., "Диагональ экрана").  
* **type** \[Enum\] \- Тип данных атрибута (см. раздел 4).  
* **description** \[String, Nullable\] \- Подсказка для пользователя.  
* **isRequired** \[Boolean\] \- Флаг обязательности заполнения.  
* **displayStyle** \[Enum: 'list', 'dropdown'\] \- Формат отображения в UI (только для списочных типов).  
* **options** \[String/Array, Nullable\] \- Возможные значения выбора (для перечислений без привязки к справочнику).  
* **dictionaryId** \[String/UUID, Nullable\] \- Ссылка на Dictionaries (только для типа reference).  
* **formula** \[String, Nullable\] \- Логика вычисления (только для типа calculated).

#### **3.4. Сущность Products (Номенклатура)**

Конечная база объектов.

* **id** \[String/UUID\] \- Уникальный идентификатор номенклатурной позиции (Primary Key).  
* **templateId** \[String/UUID\] \- Ссылка на шаблон (Foreign Key).  
* **sku** \[String\] \- Артикул или уникальный бизнес-идентификатор.  
* **name** \[String\] \- Базовое системное наименование (может генерироваться автоматически по формуле).  
* **attributeValues** \[JSON / Map / Key-Value\] \- Динамические значения атрибутов. Ключами выступают id из сущности Attributes, значениями — типизированные данные (строки, числа, массивы). Пример: {"attr\_id\_1": "Apple", "attr\_id\_2": \["Red", "Black"\], "attr\_id\_3": 599.99}.

### **4\. Справочник типов данных (Attribute Types)**

Система поддерживает следующие типы данных для построения динамических интерфейсов:

| Тип (Системный код) | Назначение | Ожидаемое значение в attributeValues |
| :---- | :---- | :---- |
| string | Короткий текст | String |
| integer | Целые числа | Number (целое) |
| float | Числа с плавающей точкой | Number (с дробной частью) |
| boolean | Логический тип (Да/Нет) | Boolean / String ("true"/"false") |
| date | Дата без времени | String (YYYY-MM-DD) |
| time | Время | String (HH:MM) |
| datetime | Дата и время | String (ISO 8601\) |
| color\_rgb | Выбор цвета (RGB Hex) | String (HEX: "\#FFFFFF") |
| enum\_radio | Одиночный выбор из заданных опций | String (выбранная опция) |
| enum\_checkbox | Множественный выбор из опций | Array of Strings |
| reference | Ссылка на справочник | String (одно из значений Dictionaries.items) |
| calculated | Расчетное поле на базе формулы | *Не сохраняется в БД (вычисляется "на лету") или кэшируется* |

### **5\. Механизм расчетных полей (Формулы)**

Формулы позволяют создавать производные значения (например, автоматическая генерация полного наименования на основе бренда, модели и объема памяти) без дублирования данных.

#### **5.1. Синтаксис и интерполяция**

Формула задается в виде строки, имитирующей синтаксис MS Excel.

Интерполяция (ссылка на другие атрибуты текущего шаблона) осуществляется через фигурные скобки: {Название Атрибута}.

**Алгоритм парсинга формулы (Псевдокод):**

1. Извлечь все вхождения паттерна {...} из строки формулы.  
2. Сопоставить имя внутри скобок с name атрибута в текущем шаблоне.  
3. Получить фактическое значение этого атрибута из поля attributeValues текущего продукта.  
4. Выполнить безопасное приведение типов:  
   * Если атрибут типа integer или float, подставить число как есть (или 0, если пусто).  
   * Если тип текстовый или список, подставить значение обернутым в кавычки с экранированием ("Значение").  
5. Передать итоговую строку в интерпретатор функций (AST-парсер).

#### **5.2. Поддерживаемые функции**

Реализация интерпретатора должна поддерживать следующий набор бизнес-функций:

**Текстовые функции:**

* СЦЕПИТЬ(текст1, текст2, ...) — объединение строк. Пример: СЦЕПИТЬ({Бренд}, " ", {Модель}).  
* ПСТР(текст, старт, количество) — извлечение подстроки.  
* ЛЕВСИМВ(текст, количество) — символы с начала строки.  
* ПРАВСИМВ(текст, количество) — символы с конца строки.  
* СЖПРОБЕЛЫ(текст) — удаление лишних пробелов.  
* ДЛСТР(текст) — длина строки.  
* ПОИСК(искомое, текст) — позиция первого вхождения подстроки.  
* ТЕКСТ(значение) — приведение любого типа к строке.

**Функции даты и времени:**

* СЕГОДНЯ() — возвращает текущую дату.  
* ТДАТА() — возвращает текущую дату и время.  
* РАЗНДАТ(дата1, дата2, "единица") — разница между датами в днях ("д"), месяцах ("м") или годах ("г").  
* ДЕНЬ(дата), МЕСЯЦ(дата), ГОД(дата), ДЕНЬНЕДЕЛИ(дата) — извлечение компонентов даты.

**Логические и математические операции:**

* Стандартные операторы: \+, \-, \*, /, \>, \<, \==.  
* ЕСЛИ(условие, значение\_истина, значение\_ложь) — условный оператор.  
* И(условие1, условие2, ...) — логическое И.  
* ИЛИ(условие1, условие2, ...) — логическое ИЛИ.  
* НЕ(условие) — логическое отрицание.

#### **5.3. Требования безопасности при расчетах**

На стороне клиента (фронтенд) и сервера (бэкенд) **категорически запрещено** использовать нативные функции типа eval() или new Function() для выполнения пользовательских формул.

Необходимо использовать специализированные AST-парсеры (Abstract Syntax Tree), например math.js или formula-parser, для предотвращения инъекций произвольного JavaScript-кода (XSS / RCE).

### **6\. API Контракты (Спецификация интеграции)**

Для работы фронтенд-виджета бэкенд целевой системы должен реализовать следующие RESTful endpoints.

#### **6.1. Dictionaries API**

* GET /api/v1/mdm/dictionaries — получение полного списка справочников.  
* POST /api/v1/mdm/dictionaries — создание нового справочника. Payload: { name: string, items: string\[\] }.  
* PUT /api/v1/mdm/dictionaries/{id} — обновление справочника.  
* DELETE /api/v1/mdm/dictionaries/{id} — удаление справочника (с проверкой на использование в таблице Attributes).

#### **6.2. Templates API**

* GET /api/v1/mdm/templates — получение списка шаблонов с вложенным массивом attributes.  
* POST /api/v1/mdm/templates — создание/обновление шаблона и его атрибутов (Upsert). Payload должен включать массив attributes.  
* DELETE /api/v1/mdm/templates/{id} — удаление шаблона (допустимо только если нет связанных Products).

#### **6.3. Products API**

* GET /api/v1/mdm/products — получение номенклатуры. Должна поддерживаться фильтрация: ?templateId={id}.  
* POST /api/v1/mdm/products — создание/обновление товара. Бэкенд должен валидировать объект attributeValues на соответствие правилам (типы данных, isRequired) схемы Attributes перед сохранением в БД.  
* DELETE /api/v1/mdm/products/{id} — удаление товара.
